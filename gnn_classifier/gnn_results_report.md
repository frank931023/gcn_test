# GCN / GraphSAGE 黑名單用戶分類報告

## 方法說明

### 資料來源
- `crypto_transfer.csv`：加密貨幣轉帳紀錄
- `train_label.csv`：51,017 筆有標籤用戶（正常 49,377 / 黑名單 1,640）

### 圖的建構方式
- **節點**：`train_label.csv` 中所有有標籤的用戶（51,017 個）
- **邊**：`crypto_transfer.csv` 中 `sub_kind==1`（平台內部轉帳），兩端都是有標籤用戶的邊，共 **1,446 條**（無向，加入反向邊）
- 圖非常稀疏：51,017 個節點，只有 1,446 條邊，平均每個節點 < 0.06 條邊

> **注意**：`relation_user_id` 欄位為 float 格式（如 `83439.0`），需先轉 int 再轉 str 才能與 `train_label` 的 user_id 對齊，這是初版腳本邊數為 0 的原因。

### 節點特徵（6 維，標準化後輸入）
| 特徵 | 說明 |
|---|---|
| `out_deg` | 出度（匯出筆數） |
| `in_deg` | 入度（收款筆數） |
| `asym` | 不對稱比 = (out-in)/(out+in) |
| `total` | 總交易次數 |
| `log1p_out` | log1p(out_deg) |
| `log1p_in` | log1p(in_deg) |

### 模型架構
兩個模型結構相同，差別只在圖卷積層：
- **GCN**：2 層 GCNConv → Linear，hidden=64，dropout=0.5
- **GraphSAGE**：2 層 SAGEConv → Linear，hidden=64，dropout=0.5

### 訓練設定
- Optimizer：Adam，lr=0.005，weight_decay=5e-4
- Epochs：200
- 不平衡處理：weighted cross-entropy，黑名單權重 ×30.1（正負比 1:30）
- Train/Test split：80/20，stratified

## 結論與問題診斷

### 為什麼結果這麼差？

**核心問題：圖太稀疏，GNN 無法發揮鄰居聚合的優勢。**

| 問題 | 說明 |
|---|---|
| 邊數極少 | 51,017 個節點只有 1,446 條邊，絕大多數節點是孤立節點，GNN 退化成純 MLP |
| 無法捕捉「周圍都是黑名單」 | 因為幾乎沒有鄰居，鄰居資訊聚合無從發揮 |
| AUC-ROC ≈ 0.5 | 等同隨機猜測，模型沒有學到有效的分類邊界 |

### 根本原因
`crypto_transfer` 中 `sub_kind==1`（內部轉帳）只有 4,473 筆，且兩端都是有標籤用戶的只有 1,446 條，圖的連通性極低。

### 改進方向

1. **擴大圖的邊**：把 `sub_kind==0` 的外部轉帳也納入（`user_id` → `from/to_wallet_hash`），讓節點透過錢包地址間接連接
2. **異質圖（Heterogeneous Graph）**：用戶節點 + 錢包節點，邊 = 轉帳關係，能大幅增加連通性
3. **退回傳統 ML**：在圖稀疏的情況下，LightGBM + degree/asym 特徵的效果會遠優於 GNN（參考 `nmf_lightgbm/` 的做法）
